# EZchain V2 进展与后续推进

本文件用于把“当前做到哪里”与“接下来先做什么”放到同一处说明。

结论先说清楚：

- `V2` 已经是本项目默认的开发、验证、演示路径。
- `V2` 已经具备本地闭环和发布判断基础，不再只是实验草稿。
- 项目仍处在从 `V1` 过渡到 `V2` 的阶段，还不能把当前状态说成“完整公开网络成品”。

这里的两层口径固定如下：

- 项目主线层面：`V2` 已经成立
- 对外交付层面：仍需真实官方测试网最终确认

本文件以当前仓库可见实现、`doc/PROJECT_CHECKPOINT_2026-03-20.md`、`doc/V2_DEFAULT_READINESS.md`、`doc/MVP_ROADMAP_NEXT.md` 为依据整理。

## 1. 当前进展

### 1.1 已经完成的主线能力

当前仓库已经形成一条清晰的 V2 主线：

- `EZ_V2/`
  - 协议对象、编码、签名、状态、验证器、钱包存储、运行时、本地网络能力已独立成线。
- `EZ_App/`
  - CLI、本地服务、节点启停、配置解析、钱包入口已接到 V2 路径。
- `configs/`
  - 已提供本地 V2 和官方测试网配置模板。
- `EZ_Test/`
  - 已提供 V2 协议、钱包、运行时、本地网络、应用层、节点管理、acceptance 测试。
- `scripts/`
  - 已提供发布门禁、稳定性门禁、测试网 profile 校验、外部试用记录和 readiness 判断工具。

换句话说，V2 现在已经不是“只有对象定义和草稿文档”，而是有了：

- 可跑的本地路径
- 可测的验收路径
- 可用于发布判断的治理路径

### 1.2 当前已经成立的判断

按仓库当前文档口径，下面这些判断已经成立：

- V2 是默认开发路径
- V2 是默认本地验证路径
- V2 是默认 acceptance 路径
- V2 是默认 RC / readiness 判断路径
- V1 进入 `legacy / freeze`

这表示今后的新工作，默认都应继续进入 `EZ_V2/`、`EZ_App/`、`EZ_Test/`、`scripts/` 这一条线，而不是回到旧的 V1 主路径继续加功能。

但这不等于可以把当前状态说成“默认正式交付路径已经完全收口”。

### 1.3 还没有完全结束的部分

当前还不能把状态说得过满，主要有四个边界：

1. 还不是完整公开网络节点栈
- README 已明确说明当前仓库还不是“finished public-network V2 node stack”。

2. 官方测试网还差一次真实可达环境的最终确认
- 当前检查点说明，在允许“当前环境可能不可达”的条件下，`release_report` 和 `v2_readiness` 已可通过。
- 但仍建议在真实可达的官方测试网环境，再做一轮不带放宽参数的正式验证。
- 单机伪远端这类 `single-host-rehearsal` 只算预演，不算最终外部证明。

3. 稳定性门禁仍需继续补强
- 当前脚本和门禁已经存在。
- 但长时间运行、抖动恢复、重复消息、异常输入这类场景，还需要更稳定、更可重复的阈值判断。

4. 外部试用与发版材料还需要继续收口
- 目标不是再堆很多功能，而是让非开发用户按文档就能跑通。
- 同时让 RC、风险记录、回滚记录和关键观测指标更统一。

## 2. 当前阶段的工作重点

如果只看接下来一段时间最该做什么，优先级建议固定为下面四项。

### P0. 稳定性门禁收口

目标：把“能跑”提升为“能稳定地过门禁”。

重点包括：

- 长时间运行用例
- 周期重启恢复
- 网络抖动与重连
- 重复消息与异常 payload
- 结构化结果输出和阈值判断

完成标志：

- 稳定性 gate 在本地和 CI 都能重复执行
- 关键红线明确
- 失败能直接阻断 RC，而不是靠人工判断

### P0. 官方测试网外部试用闭环

目标：让非开发用户按文档独立走通一次完整流程。

重点包括：

- 新机器安装验证
- 切换到 `official-testnet` profile
- 连通性检查
- 建钱包或导入钱包
- faucet、send、history、receipt 一致性

完成标志：

- 按 `doc/INSTALLATION.md` 与 `doc/USER_QUICKSTART.md`，普通试用者可在合理时间内跑通
- 至少形成一轮真实外部试用记录

### P1. 发版标准化

目标：降低发版过程中的临时性和不确定性。

重点包括：

- 固定 RC 模板
- 固定已知风险模板
- 固定回滚记录模板
- 固定最小观测指标

建议最小观测集：

- 节点在线率
- 交易成功率
- 确认延迟
- 错误码分布

完成标志：

- 每轮 RC 都能产出同构报告
- Runbook 可重复演练

### P1. 持续压缩 V1 的活跃面

目标：避免团队在过渡期重新走回旧路径。

执行原则：

- 新功能默认进入 V2 lane
- V1 仅保留兼容、对照和最小修补
- 不把新的协议能力继续加回 `EZ_VPB`、`EZ_Tx_Pool`、`EZ_Main_Chain` 这些旧模块

## 3. 推荐执行顺序

建议后续按下面顺序推进，不要并行摊太散：

1. 先做稳定性门禁收口
- 这是“V2 默认路径能不能站稳”的基础。

2. 再做真实官方测试网环境验证
- 这是“项目内部默认路径”向“更可信的对外交付路径”过渡的关键证据。

3. 再收口外部试用文档与发布模板
- 这样后续版本推进才不会每次重新组织流程。

4. 最后再考虑进一步缩减 V1 暴露面
- 这一步应建立在前面几项已经稳定的前提上。

## 4. 本阶段不做的事

为了避免方向发散，当前阶段明确不把下面这些作为主目标：

- 主网发布
- 开放式无许可节点接入
- 大规模 GUI 重构
- 为了“看起来更现代”而做大改架构
- 在 V1 路径上继续扩张新功能

## 5. 一句话总结

现在的 V2，可以理解为：

**主线已经切过去了，基础闭环也已经成了，接下来不是另起炉灶，而是把稳定性、测试网试用和发版流程收得更硬。**
