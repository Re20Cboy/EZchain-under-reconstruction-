# MVP Roadmap (Next)

本文件用于说明 V2 已成为默认主线之后，下一阶段应如何继续推进。

当前判断很简单：

- V2 已经不是“试试看”的分支，而是默认开发和默认验证路径。
- 后续重点不是另起一套新架构，而是把现有 V2 主线继续收硬、收稳、收成可反复交付。

## 执行顺序

建议按下面顺序推进，避免同时摊太多战线：

1. 先收口稳定性门禁
2. 再完成真实官方测试网外部试用闭环
3. 再固化发版模板和最小观测集
4. 持续保持 V1 freeze，不把新能力加回旧路径

## Milestone A: 稳定性门禁收口（优先级 P0）

目标：把现有 smoke/gate 从“能跑一次”提升到“能用于发布判断”。

### 要做的事

1. 扩展长稳用例
- 覆盖连续运行、周期重启、断网重连、重复消息、异常 payload。
- 产出结构化结果，例如 JSON 汇总与阈值判断。

2. 固定关键红线
- 明确失败率、恢复时间、重试次数、异常退出这类指标的可接受范围。
- 避免测试通过与否仍靠人工解释。

3. 接入发布门禁
- 将稳定性关键阈值纳入 `release_candidate.py` 或同级发布判断默认路径。
- 稳定性不过，RC 直接失败。

### 验收标准

- 稳定性 gate 在 CI 和本地都能重复执行。
- 结果输出统一，能直接用于发布报告。
- 关键阈值明确，失败时会阻断 RC。

## Milestone B: 测试网外部试用闭环（优先级 P0）

目标：让非开发用户按现有文档独立走通一次完整流程。

### 要做的事

1. 一次性安装验证
- 在 macOS 和 Windows 新环境验证安装流程。
- 完成 profile 切换、官方测试网连通性检查。

2. 钱包与交易闭环验证
- 创建或导入钱包。
- 执行 faucet、send、history、receipt 检查。
- 确保 CLI、服务接口、文档描述一致。

3. 形成外部试用记录
- 使用统一模板记录试用结果、问题、修复动作。
- 将记录纳入 readiness / release 证据链。

### 验收标准

- 按 `doc/INSTALLATION.md` 和 `doc/USER_QUICKSTART.md`，普通试用者可在 30 分钟内跑通。
- 至少形成一轮真实外部试用闭环。
- 能在真实可达环境中完成一次不带放宽参数的官方测试网验证。

## Milestone C: 发布标准化（优先级 P1）

目标：降低每次发版时的临时组织成本。

### 要做的事

1. 固定 RC 模板
- 发布说明模板
- 已知风险模板
- 回滚记录模板

2. 固定最小观测集
- 节点在线率
- 交易成功率
- 确认延迟
- 错误码分布

3. 固定报告输出
- 让每次 RC 都能生成结构相同的报告和清单。
- 让 runbook 能支撑重复演练。

### 验收标准

- 每次 RC 都能自动生成同构报告。
- 发布记录、风险记录、回滚记录格式统一。
- Runbook 能支持演练并复现问题。

## Milestone D: 持续压缩 V1 活跃面（优先级 P1）

目标：避免团队在过渡期重新依赖旧路径。

### 要做的事

1. 保持 V1 freeze
- V1 仅保留兼容、对照、最小阻断修复。

2. 新工作继续进入 V2 lane
- 协议与运行时进入 `EZ_V2/`
- CLI、服务、节点编排进入 `EZ_App/`
- 验证与发布收口进入 `EZ_Test/` 和 `scripts/`

3. 不在旧模块继续扩张新语义
- 不把新协议能力继续加回 `EZ_VPB`、`EZ_Tx_Pool`、`EZ_Main_Chain` 等旧路径。

### 验收标准

- 新增功能默认不依赖 V1 运行时对象。
- V1 保持可参考、可兼容，但不再承担主线演进责任。

## 非目标（当前阶段不做）

- 主网发布
- 开放式无许可节点加入
- 移动端钱包与复杂 GUI 重构
- 为了“统一外观”而做大规模架构重写
- 在 V1 路径上继续扩张新功能
