# EZchain V2 节点角色与 App 分层边界

## 1. 文档目的

本文件专门回答一个会长期影响实现的问题：

> 当 V2 同时存在账户节点与共识节点时，哪些逻辑属于 `EZ_V2`，哪些逻辑属于 `EZ_App`？

结论先固定：

- **节点“会做什么”属于 `EZ_V2`**
- **当前进程“启动哪一种节点”属于 `EZ_App`**

也就是说：

- `EZ_V2` 负责角色内部逻辑
- `EZ_App` 负责进程入口、配置解释、模式选择、生命周期管理

本文件是后续文件归属、模式设计、CLI 扩展和测试拆分的统一约束。

---

## 2. 先把两个概念分开

后续实现中，必须严格区分下面两件事。

### 2.1 节点角色

节点角色指“这个节点在网络里承担什么职责”。

V2 当前和后续允许的角色为：

- `consensus`
  - 负责接收 bundle、打包区块、提交区块、维护链状态、提供 receipt / chain state 查询
- `account`
  - 负责持有钱包、本地值状态、提交 bundle、接收 receipt、接收 transfer package、做本地恢复与同步
- `observer`
  - 后续可选
  - 只做状态观察、区块跟随、查询，不参与共识也不持有账户私钥

### 2.2 运行模式

运行模式指“当前这个进程怎么启动、打包了哪些角色、用来服务什么场景”。

运行模式是 App 侧概念，不是协议概念。

例如：

- 本地演示模式
- 只跑共识节点模式
- 只跑账户节点模式
- 官方测试网连接模式

同一个角色可以出现在不同运行模式中。

例如：

- `consensus` 可以出现在 `v2-localnet` 中
- `consensus` 也可以出现在“只跑共识节点”的独立模式中

所以后续绝不能把：

- “角色”
- “模式”

混成同一个概念。

---

## 3. 分层总规则

### 3.1 `EZ_V2` 的职责

`EZ_V2` 负责所有“角色本身的内部能力”。

#### 共识节点内部逻辑属于 `EZ_V2`

包括但不限于：

- `bundle_submit / bundle_ack / bundle_reject`
- 区块生产与区块应用
- receipt 生成、缓存、查询、分发
- chain state 查询
- 区块同步请求处理
- transport 消息处理
- 共识节点本地状态持久化

#### 账户节点内部逻辑属于 `EZ_V2`

包括但不限于：

- bundle 构造与提交
- bundle pending 状态维护
- receipt 接收、校验、应用
- transfer package 接收与本地入账
- 链状态跟踪
- 节点重启后的本地恢复
- 钱包本地状态与网络同步衔接

#### 角色宿主也属于 `EZ_V2`

像当前已经出现的：

- `V2ConsensusHost`
- `V2AccountHost`

这类对象，本质上是在表达“角色内部运行逻辑”，因此应该继续放在 `EZ_V2`，而不是搬到 `EZ_App`。

### 3.2 `EZ_App` 的职责

`EZ_App` 负责所有“当前进程怎么跑”的问题。

包括但不限于：

- CLI 参数与子命令
- 配置文件解析
- 根据配置选择运行模式
- 启动哪个进程
- 停止哪个进程
- 本地 service API
- 本地节点状态展示
- 本地开发和运维入口

用大白话说：

- `EZ_V2` 负责发动机和变速箱
- `EZ_App` 负责点火、挂挡、选开哪辆车

---

## 4. 后续固定的设计口径

### 4.1 角色逻辑不因为部署方式变化而搬家

即使未来某台机器：

- 只跑共识节点
- 只跑账户节点
- 或一个进程里临时打包多个角色

也不改变角色逻辑归属。

规则是：

- 角色内部逻辑始终留在 `EZ_V2`
- 只是由 `EZ_App` 决定是否实例化该角色

### 4.2 App 层只做编排，不重写角色逻辑

`EZ_App` 可以决定：

- 启哪个角色
- 角色之间是否在同一进程
- 用哪份配置
- 用哪种运行模式

但 `EZ_App` 不应自己重新实现：

- receipt 处理
- block apply
- transfer package 接收
- 共识消息处理

否则会导致角色逻辑在两层重复，各自越走越偏。

### 4.3 模式可以组合角色，但角色边界不能反过来依赖模式

后续允许存在这样的模式：

- `v2-localnet`
  - 本地开发便利模式
  - 可以把账户侧与共识侧组合在一起
- `v2-consensus`
  - 只跑共识角色
- `v2-account`
  - 只跑账户角色
- `v2-observer`
  - 后续只观察不签名的节点

但这些模式都只是 App 层入口名字。

`EZ_V2` 内部不能写成：

- “因为当前模式叫 `v2-localnet`，所以这是某种特殊协议角色”

正确方式应是：

- App 层把模式翻译成要启动的角色集合
- `EZ_V2` 只接收已经明确的角色实例和网络参数

---

## 5. 推荐的数据与调用方向

后续固定采用下面这条方向：

1. CLI / config 在 `EZ_App`
2. `EZ_App` 解析出运行模式
3. `EZ_App` 把运行模式翻译成角色集合
4. `EZ_App` 实例化对应的 `EZ_V2` 角色宿主
5. `EZ_V2` 宿主处理角色内部网络与状态逻辑

所以调用方向应保持为：

- `EZ_App -> EZ_V2`

而不是：

- `EZ_V2 -> EZ_App`

### 不允许的反向依赖

后续不应出现：

- `EZ_V2` 导入 `EZ_App/cli.py`
- `EZ_V2` 导入 `EZ_App/node_manager.py`
- `EZ_V2` 依赖 App 层模式字符串来决定协议语义

### 允许的关系

- `EZ_App` 导入 `EZ_V2` 的宿主、运行时、控制面
- `EZ_App` 为 `EZ_V2` 提供 config、路径、进程生命周期

---

## 6. 未来推荐的模式整理

当前仓库已有：

- `v2-localnet`
- `v2-tcp-consensus`

后续建议收敛为更直白的一组 App 模式：

- `v2-localnet`
  - 本地开发闭环模式
  - 默认可组合最小账户侧和最小共识侧
- `v2-consensus`
  - 单机只跑共识角色
- `v2-account`
  - 单机只跑账户角色
- `official-testnet`
  - 连接外部既有测试网
  - 本地更像账户侧入口，而不是本地自建共识入口

其中：

- `v2-tcp-consensus` 可以视为当前过渡命名
- 后续若要对外长期保留，建议逐步收敛到 `v2-consensus`

本条目前是设计建议，不代表当前 CLI 已全部实现。

## 7. 文件归属规则

后续如果新增代码，按下面归属。

### 放在 `EZ_V2/`

- 共识节点宿主
- 账户节点宿主
- observer 宿主
- 网络消息定义
- transport 抽象与实现
- receipt / block / chain state 同步逻辑
- 角色状态恢复逻辑

### 放在 `EZ_App/`

- `node start/stop/status` 命令扩展
- 模式字符串与参数解析
- 配置校验
- 进程启动器
- 本地 API 暴露
- 本地面板展示

### 不建议新增的中间灰层

后续不要再新造一个顶层目录，专门放“介于 `EZ_V2` 和 `EZ_App` 之间的角色模式胶水层”。

原因很简单：

- 角色内部逻辑已经有 `EZ_V2`
- 启动编排已经有 `EZ_App`

中间再插一层，通常只会让边界更乱。

---

## 8. 测试边界也要跟着分开

### `EZ_V2` 测试关注

- 单个角色内部逻辑是否正确
- 网络消息处理是否正确
- 状态恢复是否正确
- receipt / chain sync 是否正确

### `EZ_App` 测试关注

- 某种模式是否能正确启动
- 配置是否被正确翻译为角色集合
- 节点进程是否能 start / stop / status
- CLI / service 输出是否正确

这样后续如果“只跑共识节点”启动失败，就先看 App 编排；
如果“共识节点收不到 bundle”，就先看 `EZ_V2` 角色逻辑。

---

## 9. 最终结论

这个问题的长期答案固定为：

**未来某台机器只跑共识节点，或只跑账户节点，主要由 `EZ_App` 决定“启动哪一种角色”；但这些角色真正做事的内部逻辑，仍然应放在 `EZ_V2`。**

再说得更直白一点：

- `EZ_V2` 负责“共识节点怎么工作、账户节点怎么工作”
- `EZ_App` 负责“这次把哪一种节点跑起来”

这条边界后续不应再反复摇摆。
